FEATURE PROPOSAL

"Text It" — SMS & XMPP Messaging Integration
Send analysis results, intelligence briefs, and reports directly to any phone number via XMPP-to-SMS gateway
Date: 2026-03-15
Author: n3r4-life
Repository: github.com/n3r4-life/argus360
Status: Proposal


1. Executive Summary

This proposal introduces a "Text It" button to Argus — a one-click SMS/messaging capability that leverages the XMPP protocol and SMS gateway infrastructure to send analysis results, intelligence briefs, Draft Pad reports, and any other Argus output directly to a phone number as a text message.

The feature requires zero new infrastructure from the developer. It uses the same privacy-first, user-configured provider model that Argus applies to every external service: the user provides their own XMPP credentials and gateway, Argus handles the connection and formatting.

XMPP (Jabber) was selected because it is federated, supports end-to-end encryption (OMEMO/OTR), has native WebSocket transport (no relay bridge required), and has a mature ecosystem of SMS gateway providers that map phone numbers to XMPP JIDs.


2. Problem Statement

Argus currently supports sharing content via X (Twitter), Reddit, LinkedIn, Email, GitHub Gist, Pastebin, and PrivateBin. These cover web-based and developer-oriented distribution channels. However, the most universal and immediate communication channel — SMS text messaging — is absent.

Many operational scenarios require pushing intelligence or alerts to individuals who are not at a computer, do not use social media, or need immediate notification on their phone. SMS reaches every phone on earth with no app install required.

Without this capability, users must manually copy content from Argus, switch to a separate messaging application, paste, and send. This breaks the workflow and introduces friction in time-sensitive situations.


3. Proposed Solution

3.1 Architecture Overview

The solution adds an XMPP messaging provider to Argus and a "Text It" share button across all content surfaces. The architecture follows the existing Argus provider pattern:

    • Settings UI: New provider tab under Data Providers for XMPP configuration (server, JID, password, gateway domain). Identical layout to existing provider tabs.
    • Connection Layer: WebSocket connection from the browser directly to the XMPP server. No relay, no companion, no external infrastructure operated by the developer.
    • Gateway Routing: Messages addressed to phone numbers are formatted as XMPP JIDs using the configured gateway domain (e.g., +15551234567@cheogram.com).
    • Share Button: "Text It" button added to the existing share toolbar on Results, Draft Pad, Chat, History, and Workbench surfaces.
    • Credential Storage: XMPP credentials stored via ArgusVault.writeSensitive(), encrypted with AES-256-GCM when Vault is enabled.

3.2 Data Flow

    1. User clicks "Text It" on any content surface (analysis result, draft, chat response, etc.).
    2. Argus prompts for a recipient phone number (or selects from Sources contacts if configured).
    3. Content is formatted for SMS: stripped to plain text, truncated if needed, with a link to the full content via configured paste service if the message exceeds SMS length limits.
    4. Argus sends a message to background.js via browser.runtime.sendMessage({ action: "xmppSend", ... }).
    5. Background opens a WebSocket connection to the user's configured XMPP server (or reuses an existing connection).
    6. Message is sent as an XMPP stanza to +{number}@{gateway-domain}.
    7. The gateway provider (JMP/Cheogram, self-hosted, or commercial) delivers the message as SMS to the recipient's phone.
    8. Delivery confirmation (if supported by the gateway) is displayed in Argus via toast.

3.3 XMPP Connection Details

XMPP supports WebSocket transport natively via XEP-0206 (BOSH) and RFC 7395 (XMPP over WebSocket). Most modern XMPP servers (Prosody, ejabberd, Openfire) expose a WebSocket endpoint. This means the browser extension can connect directly — no TCP bridge, no relay server, no companion binary.

Component       Implementation              Notes
Transport       WebSocket (wss://)          Native browser API, no extension permissions beyond existing
Authentication  SASL PLAIN over TLS         Standard XMPP auth, encrypted in transit
Message Format  XMPP stanza (XML)           Lightweight, well-defined, extensible
SMS Gateway     JID mapping                 +number@gateway.domain routes to phone network
Encryption      OMEMO (optional)            End-to-end for XMPP-to-XMPP; SMS leg is unencrypted by nature
Credential Stg  ArgusVault.writeSensitive() Encrypted with Vault AES-256-GCM (PBKDF2 key derivation, 100k iterations)

XMPP Minimal Send Stanza (Phase 1 only needs this):

    <message type="chat" to="+15551234567@cheogram.com" from="user@jabber.example.com">
      <body>Your Argus analysis is ready: https://example.com/paste/abc123</body>
    </message>

This is a single XML element. Phase 1 does NOT need a full XMPP library — just WebSocket open, SASL PLAIN auth handshake (4 XML exchanges), then <message> stanzas.


4. SMS Gateway Providers

The user chooses their own gateway provider. Argus ships with configuration templates for known providers but does not mandate or default to any specific service.

Provider            Type            Cost                Setup
JMP.chat (Cheogram) Hosted service  ~$4.99/month        Sign up, get number, configure JID in Argus
sms-over-xmpp       Self-hosted    Free (+ SIM/modem)   Go binary with GSM modem or Twilio backend
SMStoXMPP           Self-hosted    Free (+ Android)     PHP gateway using Android phone as SMS bridge
Twilio + XMPP       Commercial API Pay-per-message      Twilio account + custom bridge script
Vonage (Nexmo)      Commercial API Pay-per-message      Vonage account + XMPP component

For the self-hosted options, users can leverage the Argus Terminal (ssh/ssh.html) or Argus+ companion to manage the gateway process on their own infrastructure, keeping the entire message pipeline under their control.

5.3 Send Flow

When the user clicks "Text It":
    1. A compact modal appears (same style as the paste picker in argus-chat.js showPastePicker()) with:
       - Phone number input (tel type, E.164 format)
       - Recent recipients dropdown (from Sources store, sorted by addresses[].lastUsed)
       - Message preview (plain text, character count, segment count)
    2. User enters or selects a number. Argus formats it with the configured country code.
    3. Preview shows the message that will be sent, including truncation notice if applicable.
    4. User confirms. Background handler sends via XMPP. Toast shows success/failure.
    5. If the message exceeds 1600 characters (10 SMS segments), Argus offers to auto-paste the full content to the configured paste service and send a short message with the link instead.

The modal is implemented in shared/text-it.js as a reusable component (TextIt.open(content)) that can be called from any surface. It handles the background message passing internally.


6. Contacts — Use Existing Sources Store (NOT a new store)

The proposal originally described a separate contacts list. This is unnecessary. Argus already has the Sources store (lib/storage-db.js, ArgusDB.Sources) which is a curated directory of people, handles, and contact info.

EXISTING Sources schema:
    {
      id: string,
      name: string,
      aliases: string[],
      tags: string[],
      location: string,
      notes: string,
      addresses: [{ label: string, value: string }],
      createdAt: timestamp,
      updatedAt: timestamp
    }

Sources already supports:
    - Multiple addresses per contact (label: "Phone", value: "+15551234567")
    - Tags for categorization (tag with "sms-contact" for quick filtering)
    - Full-text search across name, aliases, tags, addresses
    - CRUD via ArgusDB.Sources.save(), .getAll(), .search()
    - Included in cloud backups and Vault encryption

For the Text It feature:
    - Add a "sms" tag convention to identify SMS-capable contacts
    - Add lastUsed tracking to addresses for recent-recipients sorting
    - Show sources with phone-type addresses in the recipient picker
    - After sending, update the address.lastUsed timestamp
    - "Save contact" option in send modal saves to Sources with phone address + sms tag

No new IndexedDB store. No schema changes. Just convention + a minor field addition (lastUsed on addresses).


7. XMPP Chat Panel (Phase 2)

The "Text It" button is Phase 1 — outbound messaging. Phase 2 introduces a dedicated XMPP Chat Panel for bidirectional communication.
    • Inbound messages: Received XMPP messages (including SMS replies via gateway) appear in a chat panel within Argus.
    • Contact presence: Online/offline status for XMPP contacts.
    • Group chat: MUC (Multi-User Chat) support for XMPP rooms.
    • Automation triggers: Incoming messages can trigger automations via AutomationEngine.run() with trigger type "xmpp-message" (same pattern as trawl-data triggers).
    • AI integration: The same natural language AI layer available in Chat can assist in composing and interpreting messages.
    • OMEMO encryption: End-to-end encryption for XMPP-to-XMPP conversations (SMS leg remains unencrypted by nature of the phone network).
    • At this point, upgrade from custom minimal client to stanza.js or xmpp.js for full protocol support.


8. Privacy & Security Compliance

This feature adheres to Argus's core privacy architecture and Mozilla AMO policies:

Principle                Implementation
No developer servers     User provides their own XMPP server and gateway. Argus connects directly.
User-initiated only      Messages sent only when user explicitly clicks "Text It" and confirms.
Credential security      XMPP password via ArgusVault.writeSensitive("xmpp-password", ...). In-memory key only when Vault is unlocked.
Data minimalism          No message history stored by default. Opt-in message log for chat panel (Phase 2).
Transparency             XMPP connection status visible in UI. All transmissions user-initiated and user-visible.
AMO compliance           No new required permissions. XMPP WebSocket uses existing <all_urls> permission.
Private browsing         XMPP features disabled in private/incognito mode. No data leaks across sessions.
Wipe Everything          XMPP credentials, contacts, and message logs cleared with existing wipe function.

9.3 Background Message Handler Skeleton

    // ── XMPP / Text It ──
    if (message.action === "xmppGetStatus") return handleXmppGetStatus();
    if (message.action === "xmppTestConnection") return handleXmppTestConnection(message);
    if (message.action === "xmppSend") return handleXmppSend(message);
    if (message.action === "xmppDisconnect") return handleXmppDisconnect();

    // In handleXmppSend:
    async function handleXmppSend(message) {
      const { to, body } = message; // to = phone number, body = text content
      if (!to || !body) return { success: false, error: "Missing recipient or body" };
      try {
        // Ensure connected
        if (!_xmppWs || _xmppWs.readyState !== WebSocket.OPEN) {
          await xmppConnect(); // reads config from storage, opens WS, SASL auth
        }
        // Format JID
        const { gateway } = await getXmppConfig();
        const jid = XmppSms.formatJid(to, gateway); // "+15551234567@cheogram.com"
        const truncated = XmppSms.truncateForSms(body); // handles paste-and-link if needed
        // Send
        await XmppClient.sendMessage(_xmppWs, jid, truncated);
        // Update Sources lastUsed
        await updateSourceLastUsed(to);
        return { success: true, to: jid, chars: truncated.length };
      } catch (e) {
        return { success: false, error: e.message };
      }
    }


9.4 XMPP Auth Handshake (Phase 1 — minimal, no library)

    // 1. Open WebSocket
    ws = new WebSocket("wss://xmpp.example.com:5281/ws", "xmpp");

    // 2. Send stream open
    ws.send('<open xmlns="urn:ietf:params:xml:ns:xmpp-framing" to="jabber.example.com" version="1.0"/>');

    // 3. Server responds with stream features (including <mechanisms>)
    // 4. Send SASL PLAIN auth
    const authStr = btoa("\0" + jidUser + "\0" + password); // \0user\0pass base64
    ws.send('<auth xmlns="urn:ietf:params:xml:ns:xmpp-sasl" mechanism="PLAIN">' + authStr + '</auth>');

    // 5. Server responds <success/> or <failure/>
    // 6. Re-open stream, bind resource
    // 7. Ready to send <message> stanzas

Total: ~80-100 lines for a working send-only XMPP client. Reference pattern: ssh/ssh.js WebSocket connection with binary protocol handling.

10. Broader Communications Roadmap

The "Text It" feature is the first step in a larger communications layer for Argus. The XMPP infrastructure established here serves as the foundation for additional protocols:

Protocol        Transport                   UI                          Use Case
SMS (via XMPP)  WebSocket (native)          Text It button + Chat       Universal reach to any phone
XMPP Direct     WebSocket (native)          Chat Panel                  Encrypted messaging between XMPP users
IRC             WebSocket (native or relay)  Chat Panel                 Community channels, real-time intel monitoring
Telnet/BBS      WebSocket relay             Terminal (ssh/ssh.html)     Legacy bulletin board access, retro networks
FTP             WebSocket relay             Terminal or File Panel      Remote file management

All protocols follow the same privacy model: user-configured connections, user-provided credentials, no developer-operated infrastructure, Vault encryption for stored credentials, full audit trail for AI-initiated actions.


11. Risks & Mitigations

Risk                                Likelihood  Impact  Mitigation
SMS gateway costs surprise users    Medium      Low     Clear documentation of per-message costs. Free self-hosted options featured.
XMPP server compatibility issues    Medium      Medium  Test against Prosody, ejabberd, Openfire. Provide troubleshooting guide.
Users store sensitive intel in SMS   Medium      High    Warn SMS is unencrypted. Recommend XMPP direct for sensitive content.
AMO review questions                Low         Low     No new permissions. User-initiated. Privacy doc updated.
Message delivery failures           Medium      Low     Clear errors. Retry option. Clipboard fallback if XMPP fails.
WebSocket blocked by corp firewall  Medium      Low     Clear error message. Suggest alternative port (443). BOSH fallback for Phase 2.
Vault locked when sending           Medium      Low     Prompt user to unlock Vault. Don't fail silently.


12. Success Criteria

    • User can configure an XMPP provider in Settings and successfully test the connection.
    • User can send a text message from any content surface with three clicks or fewer (Text It → enter number → confirm).
    • Messages arrive on the recipient's phone within 30 seconds under normal conditions.
    • Long messages automatically fall back to paste-and-link with no user intervention beyond confirmation.
    • Credentials are encrypted at rest when Vault is enabled.
    • Feature degrades gracefully: if XMPP is not configured, the Text It button is hidden (not broken).
    • No new permissions required in manifest.json.
    • PRIVACY.md accurately documents all data flows.
    • Recipients saved to Sources store are searchable and appear in the recent-recipients picker.


13. Conclusion

The "Text It" feature extends Argus's reach from the browser to the most universal communication channel on earth. It does so without compromising the project's core privacy principles, without requiring developer-operated infrastructure, and without adding unnecessary complexity to the codebase.

By building on XMPP, this feature establishes the foundation for a broader communications layer that will eventually encompass encrypted messaging, IRC, and integration with the terminal-based protocols (BBS, FTP) already under development. Each addition follows the same pattern: user-configured, user-controlled, privacy-first.

Argus doesn't just analyze intelligence — it delivers it.
